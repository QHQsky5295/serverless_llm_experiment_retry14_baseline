# 对比实验日志

本文档用于记录与其他系统进行公平对比时的实验背景、源码核查、论文核查、日志排查、根因分析与结论。后续每新增一个 baseline，或对同一 baseline 进行新一轮关键排查，都应继续在本文档中追加记录。

## 记录原则

1. 所有正式结论必须同时参考三类证据：
   - 对方论文或官方文档
   - 对方真实源码或本地复现代码
   - 本地运行日志、结果 JSON 与共享 artifact
2. 不允许只依据某一次结果数值就下结论。
3. 不允许用估计值替代可真实观测的指标。
4. 如果某项指标在 baseline 中拿不到真实值，应明确记为 `null`，不能用近似值掩盖。
5. 对比实验中的数据输入必须一致：
   - 同一基座模型
   - 同一 shared trace artifact
   - 同一 shared LoRA subset artifact
   - 同一指标语义
6. 主实验背景始终围绕论文问题本身，不为 baseline 单独改题。

## 2026-04-15：ServerlessLLM 公平对比排查记录

### 1. 本轮背景

目标系统：

- 本系统：`FaaSLoRA`
- 对比系统：`ServerlessLLM`

当前正式问题背景：

- `many-LoRA concurrent serving`
- `serverless inference`
- 同一轮共享合成负载
- 所有请求都绑定 LoRA

当前重点模型：

- `Llama-2 7B`
- `Llama-2 13B`

### 2. 当前统一的公平性约束

当前正式对比中，以下约束已经固定：

1. 主场景必须是 `many-LoRA`，而不是普通单模型 serving。
2. 所有请求必须带 LoRA，即 `100% LoRA requests`。
3. 两个系统共用同一份 `trace artifact`。
4. 两个系统共用同一份 `LoRA subset artifact`。
5. 主指标统一为：
   - `TTFT_overall`
   - `TPOT`
   - `E2E`
   - `Tok/s`
   - `Cost/req`
   - `CE`
   - `SLO`

### 3. 为什么正式主实验必须使用 `100% LoRA requests`

结论：对当前论文主问题而言，`100% LoRA` 是最合理的正式主设置。

原因如下：

1. 当前论文的研究对象不是普通 LLM serving，而是 `many-LoRA` 并发 serving。
2. 如果混入无 LoRA 请求，会显著稀释：
   - LoRA 路由压力
   - LoRA 加载与缓存压力
   - GPU-ready 命中压力
   - 扩容后首次服务压力
3. 这会让对比结果偏向通用 serving，而弱化本论文真正的系统贡献。

补充结论：

- `LoRA/非 LoRA 混合流量` 可以作为后续补充实验
- 但不能取代当前正式主实验

### 4. 论文、源码与本地日志交叉结论

#### 4.1 ServerlessLLM 论文与当前本地场景并不等价

从 `ServerlessLoRA` 论文页面与相关预览信息可知，论文中对 `ServerlessLLM` 的对比场景明显比当前本地正式场景更轻：

- LoRA 数量更少
- many-LoRA 压力更低
- 硬件条件更强
- 并不是当前这种 `500 requests + 500 adapters + 100% LoRA requests` 的正式 many-LoRA 压测

因此，论文中 baseline 的绝对数值，不能直接当作本地绝对预期。

#### 4.2 ServerlessLLM 官方 many-LoRA 路径本身受限

本地官方文档与源码显示：

- LoRA 服务支持主要落在 `transformers backend`
- `vllm backend` 在当前 `Llama-2 + publicmix LoRA` 组合上不是可靠正式路径

本地 probe 现象包括：

- 7B 上出现 `all-zero token ids`
- 13B 上出现 `StreamInput must be either an integer or a list of integers`

因此，当前正式 baseline 策略是：

- 优先尝试 `vllm`
- 正确性 probe 失败后自动回退 `transformers`

这不是“照顾 baseline”，而是避免用错误路径得出无效结果。

#### 4.3 当前场景会真实放大 ServerlessLLM 的排队与协调问题

本地 `all-LoRA` 7B 正式 trace 统计为：

- 请求数：`500`
- 到达窗口：约 `64.5s`
- 平均到达率：约 `7.75 req/s`
- 平均输入：约 `1014.88 tokens`
- 平均输出：约 `150.79 tokens`
- 唯一 LoRA：`86`
- 无 LoRA 请求数：`0`

对应的历史异常结果显示：

- `TTFT` 极大
- `runtime_ttft` 却很小
- `serverless_overhead` 与 `queue_wait` 极大

这说明问题的主因不是 decode 慢，而是：

- 实例协调
- 排队等待
- serverless 路径开销

### 5. 关于“ServerlessLLM 为什么会特别差”的分层结论

#### 第一层：早期复现配置偏差

历史上我们曾经把 baseline 跑成：

- `min_instances=0`
- `keep_alive=0`

这会人为夸大冷启动和等待时间。该问题已修复。

另外，本地复现过程中还出现过一次“栈已就绪但外层检测脚本误判失败”的工程性问题：

- `start_serverlessllm_stack.sh` 在等待 `Ray runtime started.` 时硬依赖 `rg`
- 非交互环境下 `rg` 不在 PATH 中
- 导致 head 实际已经启动，但检测脚本持续报 `rg: 未找到命令`，最终超时报错

该问题已修复为：

- 优先使用 `rg`
- 若 `rg` 不可用，则自动回退 `grep -E`

因此，后续再出现“Ray 实际已启动但脚本判定失败”的概率已被显著降低。

#### 第二层：当前正式场景本身更严苛

当前本地正式场景，是围绕论文问题背景构造的 many-LoRA 并发场景，负载强度明显高于其论文中展示的 baseline 场景。

因此，如果 baseline 在这里暴露明显排队问题，这是合理现象，不应简单理解为“复现错误”。

#### 第三层：其系统结构在当前场景下的真实短板

从本地日志与 replay 结果看，`ServerlessLLM` 的瓶颈主要在：

- queue wait
- serverless overhead
- 实例路径协调

而不是：

- 模型纯 decode 性能
- 指标脚本计算错误

因此，当前结论是：

- 这个队列问题不是纯粹的复现 bug
- 它也不是完全脱离场景的固定常数
- 它是该系统在当前 `many-LoRA` 问题背景下暴露出的真实结构性瓶颈，早期又叠加了我们已经修掉的配置问题

### 6. FaaSLoRA 是否被 baseline 相关修改污染

本轮重点检查后，当前结论是：

- `FaaSLoRA` 仓库整体处于持续开发状态，不能宣称全仓库完全无历史修改
- 但 baseline 对齐工作没有改坏 `FaaSLoRA` 主指标语义
- 当前主指标定义仍保持为：
   - `TTFT_overall`
   - `TPOT`
   - `E2E`
   - `CE`

并且：

- 共享 trace 导出逻辑已强制 many-LoRA 公平约束
- `lora_request_ratio=1.0` 已被写进正式对比链路

### 7. 当前总判断

当前不能简单说：

- “ServerlessLLM 很差只是因为我们跑坏了”

也不能简单说：

- “FaaSLoRA 已经天然完胜，一切都无需再检查”

更准确的判断是：

1. 早期 baseline 确实有配置与路径问题，已经修复。
2. 修复之后，当前 many-LoRA 场景依旧会真实暴露 `ServerlessLLM` 的排队与协调瓶颈。
3. `FaaSLoRA` 在该问题背景下更有优势，主要原因并不是指标口径，而是系统设计本身更贴近 many-LoRA serverless inference。

## 2026-04-15：研究价值与基线选择补充结论

### 1. 为什么当前 many-LoRA serverless 问题仍然有研究价值

联网核查后，当前结论是：该问题不仅仍然存在，而且是当前大模型部署环境中的真实系统问题。

主要依据如下：

1. `ServerlessLLM`（OSDI 2024）表明，通用大模型的 serverless 推理仍然需要重点解决冷启动、checkpoint 加载和启动调度问题。
2. `Punica`（MLSys 2024）与 `S-LoRA`（MLSys 2024）表明，多 LoRA 并发 serving 已经成为独立且重要的系统问题，不能再被简单视为普通 LLM serving 的一个小变体。
3. 当前主流生产推理引擎和平台已经直接支持多 LoRA serving：
   - `vLLM` 官方文档支持按请求加载 LoRA，并支持运行时动态加载/卸载
   - `LoRAX` 官方文档明确支持按请求动态加载 LoRA，并可对多种不同 LoRA 做高吞吐批处理
   - `TGI` 文档也已经支持一次加载多个 LoRA adapter

这说明：

- 多 LoRA 服务不是“假设中的未来需求”
- 而是已经进入真实推理引擎与平台能力边界的当前问题

### 2. ServerlessLLM 的优点与边界

当前对 `ServerlessLLM` 的客观判断是：

#### 优点

1. 它是高质量、已发表的 serverless LLM 系统论文，研究问题明确。
2. 它的设计重心非常清晰：
   - checkpoint locality
   - 多层 checkpoint loading
   - live migration
   - startup-time optimized scheduling
3. 在“通用大模型 serverless 启动与迁移”这一问题上，它依然有代表性。

#### 边界

1. 它并不是围绕 `many-LoRA` 场景设计的。
2. 在当前本地复现路径上，它对 LoRA 的正式可用支持主要落在 `transformers backend`。
3. 对当前 `Llama-2 + publicmix LoRA` 组合，`vllm` 不是稳定正式路径。
4. 它在当前 `many-LoRA + 长 prompt + 高到达率` 场景下暴露出明显的队列与协调瓶颈。

因此：

- `ServerlessLLM` 适合作为“通用 serverless LLM baseline”
- 但不适合作为“many-LoRA 主基线”的唯一候选

### 3. ServerlessLLM 当前是否还能作为论文基线

可以，但定位必须调整。

当前更合理的定位是：

- `次要对比基线`
- `serverless 通用基线`
- `反映通用 serverless LLM 系统在 many-LoRA 场景下的不足`

而不是：

- 你的唯一主对比对象
- many-LoRA 问题上的最强专门 baseline

### 4. 当前更合理的对比对象选择

#### 第一层：主对比系统

这些系统与论文问题最接近，应优先考虑复现和对比：

1. `Punica`（MLSys 2024）
   - 关键词：`Multi-Tenant LoRA Serving`
   - 特点：明确面向多 LoRA 共享基座 serving
2. `S-LoRA`（MLSys 2024）
   - 关键词：`Serving Thousands of Concurrent LoRA Adapters`
   - 特点：明确面向海量并发 LoRA adapter serving

#### 第二层：次要系统

这些系统与论文问题部分重合，但不是 many-LoRA 专门系统：

1. `ServerlessLLM`（OSDI 2024）
   - serverless 通用基线
2. `MuxServe`（ICML 2024）
   - 多模型 serving / multiplexing 基线
3. `Aegaeon`（SOSP 2025）
   - 并发 LLM serving / GPU pooling 方向

#### 第三层：文献参考或工程参考

1. `ServerlessLoRA`（2025 预印本）
   - 问题背景非常接近
   - 但当前更适合做“次要对比文献/讨论对象”
2. `vLLM` / `LoRAX` / `TGI`
   - 更偏工程基线或生产系统参考
   - 不宜替代已发表研究系统

### 5. 论文叙事建议

当前更稳妥的论文叙事是：

1. 你的问题不是“通用 LLM serving”
2. 你的问题是“many-LoRA serverless inference”
3. 因此主贡献不应表述为“普适地优于所有 LLM serverless 系统”
4. 更准确的表述应是：
   - 在 many-LoRA serverless 场景下
   - 通用 serverless LLM 系统会出现额外冗余、LoRA 加载、资源争用与扩缩容首次服务问题
   - `FaaSLoRA` 针对这一特定场景给出了系统级优化

## 2026-04-15：Llama-2 7B `alllora_v5` 正式结果详细分析

### 1. 本轮实验身份

- 模型：`Llama-2 7B`
- 运行标签：`llama2_7b_r500_a500_seed42_alllora_v5`
- 共享 trace：`/home/qhq/serverless_llm_baselines/results/shared_rounds/llama2_7b_r500_a500_seed42_alllora_v5_trace.json`
- 共享 LoRA subset：`/home/qhq/serverless_llm_baselines/results/shared_rounds/llama2_7b_r500_a500_seed42_alllora_v5_adapter_subset.json`
- 对比结果：`/home/qhq/serverless_llm_baselines/results/comparisons/llama2_7b_r500_a500_seed42_alllora_v5.json`

本轮输入公平性确认如下：

1. 两个系统使用同一基座模型。
2. 两个系统使用同一份 shared trace artifact。
3. 两个系统使用同一份 shared LoRA subset artifact。
4. 本轮请求数为 `500`，选用 LoRA 数为 `500`。
5. 所有请求均绑定 LoRA。

### 2. 共享负载画像

本轮 shared trace 的关键特征为：

- 请求总数：`500`
- 到达窗口：约 `64.52s`
- 平均到达率：约 `7.75 req/s`
- 平均输入长度：约 `1014.88 tokens`
- 平均输出长度：约 `150.79 tokens`
- 实际命中的唯一 LoRA：`86`

这说明本轮并不是“形式上 500 个 LoRA、实际上只在少数 adapter 上反复命中”的极端简化场景，而是带有明显 LoRA 热度迁移和真实缓存压力的 many-LoRA 动态负载。

### 3. 主指标口径说明

当前正式主指标定义如下：

1. `TTFT_overall`：请求到达/提交到首个生成 token 的时间。
2. `TPOT`：首个 token 之后，平均每个输出 token 的生成时间。
3. `E2E`：请求到达/提交到完整响应结束的时间。
4. `Tok/s`：总输出 token 数除以实验总耗时。
5. `SLO`：满足 `TTFT <= 5000ms` 的请求比例。
6. `CE`：`1 / (avg_e2e_s * avg_cost_usd)`。

需要特别强调：

- 延迟均值仅在成功请求上计算。
- 因此，如果某个系统失败率很高，仅看平均 `TTFT/E2E` 依然可能低估其真实问题。
- 正式对比时必须同时看成功率、`SLO` 和主延迟指标。

### 5. 为什么这些指标会呈现这样的结果

#### 5.1 `TTFT_overall`

`FaaSLoRA` 为 `1.42s`，`ServerlessLLM` 为 `298.24s`。  
本轮最关键的结论不是“ServerlessLLM 生成慢”，而是“ServerlessLLM 在真正开始生成之前已经被队列与协调路径拖垮”。

本地 replay 结果显示：

- `ServerlessLLM avg_runtime_ttft_ms ≈ 46.19ms`
- `ServerlessLLM avg_queue_wait_ms ≈ 297800.75ms`

这说明绝大多数 `TTFT` 并不是耗在 backend decode，而是耗在：

- request queue
- instance assignment
- adapter-ready 之前的协调路径

因此，当前 many-LoRA 场景下，`TTFT` 的主要差异来自系统级等待，而不是模型本身算力。

#### 5.2 `TTFT_comp` 与 `TTFT_warm`

`FaaSLoRA` 的 `TTFT_comp / TTFT_warm` 分别约为 `953ms / 916ms`，而 `ServerlessLLM` 两者都约为 `395s`。

这说明即使把请求限制到“更可比、更接近 warm-path 的子集”，`ServerlessLLM` 仍然会被前置排队拖成分钟级。  
也就是说，它的问题不只是冷启动，而是“adapter 已近似 ready 时，系统级 serverless 路径仍然没有把 ready 转化成低 TTFT 服务”。

#### 5.3 `TPOT`

`FaaSLoRA` 为 `80.23ms`，`ServerlessLLM` 为 `131.91ms`。

这一项说明：

1. 你的系统不只在请求前半段更优。
2. 在 decode 阶段，`FaaSLoRA` 也更快。

这里的原因既包括系统路径差异，也包括正式可用 backend 差异：

- `FaaSLoRA` 走的是 many-LoRA 优化过的主 serving 路径
- `ServerlessLLM` 在当前 `Llama-2 + publicmix LoRA` 组合下只能稳定回退到 `transformers`

#### 5.4 `E2E`

`FaaSLoRA` 为 `20.79s`，`ServerlessLLM` 为 `318.38s`。  
这一项基本与 `TTFT` 结论一致：主差异并不来自生成本身，而是服务前半段排队和协调成本。

#### 5.5 `Tok/s`

表面上看，`ServerlessLLM Tok/s = 45.90`，高于 `FaaSLoRA Tok/s = 30.70`。  
但这一项单独看具有误导性，因为：

- `ServerlessLLM` 只成功完成了 `191` 个请求
- 其余 `309` 个请求在 `600s` 客户端读超时前未返回

因此，正式比较时必须结合成功率和 `SLO`。

更合理的可用吞吐解释是：

- `FaaSLoRA SLO-goodput Tok/s ≈ 28.37`
- `ServerlessLLM SLO-goodput Tok/s ≈ 0.24`

也就是说，从“真正按时、按要求完成服务”的角度看，`FaaSLoRA` 明显更强。

#### 5.6 `ColdStart`

这一项反而是 `ServerlessLLM` 更好：`7.28s` 对 `37.92s`。

这与其论文技术重点是一致的。`ServerlessLLM` 的优势更多落在：

- 大模型 serverless checkpoint 启动
- 模型加载与实例迁移

而不是 many-LoRA adapter readiness 本身。  
这说明它不是“完全没有优点”，而是“优点不在本文的核心问题上”。

#### 5.7 `ScaleUpAffected` 与 `ScaleUpFirstService`

这两项是当前最能支撑本文系统贡献的证据：

- `ScaleUpAffected`：`2237.82ms` 对 `123380.69ms`
- `ScaleUpFirstService`：`357.19ms` 对 `10997.32ms`

这说明你的系统在扩容之后更快把“物理扩容”转化成“可服务扩容”，而 baseline 在实例起来之后，仍然需要更长时间才能真正承接 LoRA 请求。

#### 5.8 成功率与失败性质

本轮 `ServerlessLLM` 失败的 `309` 个请求并不是立即返回 `500`，而是统一表现为：

- `HTTPConnectionPool(...): Read timed out. (read timeout=600.0)`

并且：

- 成功请求主要集中在到达时间约 `0s ~ 25s`
- 失败请求从到达时间约 `24.39s` 开始大面积出现
- 成功请求中的最后一批，其 `queue_wait_ms` 已接近 `581s`

因此，本轮“失败”更准确地说是：

- baseline 在当前负载下发生严重排队
- replay 客户端在 `600s` 读超时前仍未等到响应
- 于是被记录为失败

### 6. 关于“服务失败问题”是否应该修

结论：应该修，但修复目标必须定义清楚。

当前最合理的修复目标不是：

- 去给 `ServerlessLLM` 增加新的 LoRA-aware 系统机制
- 或为了让它变好而改变题目、数据、LoRA 压力

而是：

- 去掉 replay 侧 `600s` 读超时对正式结果的截断
- 让 baseline 把“真实但很慢”的结果完整跑出来

这是公平的，因为：

1. 它不改变模型、不改变 shared trace、不改变 shared LoRA subset。
2. 它不改变 baseline 的系统设计，只改变我们对其完成情况的观测窗口。
3. 它不会影响 `FaaSLoRA`、基座模型、数据集和主实验链路。

因此，本轮后续采取的 baseline-only 修复是：

- 将 `run_serverlessllm_fair_experiment.sh` 的默认 replay 超时从 `600s` 提高到 `3600s`

该修复的目标不是“让 ServerlessLLM 变快”，而是：

- 避免还在队列中的请求被客户端过早判死
- 让正式对比先从“观测被截断”变成“真实结果完整返回”

### 7. 当前阶段性判断

1. `FaaSLoRA` 在当前 many-LoRA 问题背景下，已经表现出明显系统性优势。
2. `ServerlessLLM` 的当前主要问题并不是 decode 性能，而是 queue / coordination / serverless overhead。
3. `ColdStart` 仍然是 `ServerlessLLM` 的真实强项，这部分值得在论文中客观呈现。
4. `ServerlessLLM` 仍可保留为“通用 serverless LLM 基线”，但不应成为唯一主基线。
5. 如果后续继续补 baseline，优先级更高的应是：
   - `Punica`
   - `S-LoRA`

### 4. 结合论文与代码实现的根因分析

#### 4.1 论文目标与当前问题背景并不一致

`ServerlessLLM` 论文的核心贡献是：

1. 多层 checkpoint 加载
2. 运行时 live migration
3. 启动时间优化的 model scheduling

其设计中心是通用大模型 serverless inference 中的：

- checkpoint locality
- model startup latency
- instance migration

而不是当前论文的 many-LoRA serverless inference 问题。

这与当前结果是吻合的：  
它的 `ColdStart` 仍然相对较好，但 `TTFT` 主体却被 many-LoRA 条件下的 queue/coordination 拖垮。

#### 4.2 当前代码路径不是 LoRA-aware 路由

本地 `RoundRobinRouter` 实现显示：

- 请求首先进入全局 `asyncio.Queue`
- load balancer 按 `round-robin` 方式选实例
- 选实例时只看 queue capacity，不看 adapter 命中或 LoRA ready 状态

也就是说，这条正式路径并不做：

- adapter affinity routing
- hot-adapter aware placement
- LoRA-aware warmup

这与 `FaaSLoRA` 的系统目标根本不同。

#### 4.3 `transformers` LoRA 路径是“请求分配后再加载”

当前正式可用路径下，`ServerlessLLM` 会在请求已经分配到实例之后，再执行：

- `load_lora_adapter(...)`

如果该实例上尚未加载对应 adapter，则会发生真实加载。  
这意味着：

1. LoRA ready 不是路由前状态，而是请求落到实例后的后处理。
2. 请求即使已进入 ready instance，也可能因为 target adapter 未就绪而继续等待。

这正是 many-LoRA 场景中的关键不匹配来源。

#### 4.4 当前主瓶颈不是 decode，也不是冷启动

新结果中：

- `Runtime_TTFT_avg_ms ≈ 73.15ms`
- `Cold_start_avg_ms ≈ 7767.91ms`
- `TTFT_overall_avg_ms ≈ 715709.07ms`
- `avg_queue_wait_ms ≈ 715137.64ms`

这说明：

- decode 不是主要瓶颈
- cold start 不是主要瓶颈
- 主要瓶颈就是 queue wait 与 serverless overhead

因此，当前 baseline 的核心问题不是“模型慢”，而是“在当前 many-LoRA 负载下，它的请求组织方式和 adapter-ready 路径不适合这个问题”。

#### 4.5 在当前公平约束下，参数调节空间已经接近耗尽

当前正式 deploy 已经满足：

- `min_instances=1`
- `max_instances=4`
- `keep_alive=10`
- 4 个实例都真实拉起并承接了请求

这说明问题已经不是：

- 没扩出来
- keep_alive 太激进
- min/max instance 配错

而是在 4 张 GPU 都已经被用满的情况下，系统仍然无法把 many-LoRA 请求组织成低等待服务。

### 5. 是否还能在我们的原则下把它拉回“可比范围”

结论：基本不能，至少不能通过“只调官方参数、不改系统设计”的方式做到。

原因如下：

1. 目前已经证明，扩大观测窗口之后它可以 `500/500` 完成服务。
2. 但主瓶颈来自系统设计与问题背景的不匹配，而不是单一脚本 bug。
3. 如果继续想把它拉回可比范围，只剩下两条路：
   - 降低 LoRA 压力、降低到达率、换更强硬件
   - 给它加入 LoRA-aware 路由、adapter 预热或更强 batching
4. 前者会改变题目和正式场景，不符合公平原则。
5. 后者会改动 baseline 核心设计，不再是 `ServerlessLLM`。

因此，在当前原则下，`ServerlessLLM` 最多只能被修到：

- “能完整服务”

但很难被修到：

- “在当前正式 many-LoRA 场景下进入与你系统相近的性能区间”

### 6. 当前定位建议

基于这轮结果，当前对 `ServerlessLLM` 的定位应当进一步收紧为：

- 它可以保留为“通用 serverless LLM 基线”
- 它不适合充当 many-LoRA 问题的唯一主基线
- 在本文中，它更适合回答：
  - 通用 serverless LLM 系统在 many-LoRA 场景下会暴露什么问题

而真正更匹配的问题基线，应优先考虑：

- `Punica`
- `S-LoRA`

- 这层解释会在 `Qwen` 长上下文边界上引入和 `FaaSLoRA` tokenizer 预算不一致的偏差

因此，这次修复不是继续补一个兜底裁剪，而是把根因直接去掉：

- 不再走 `/v1/completions`
- 正式切到 `SGLang` 原生 `/generate`
- 请求体中直接发送和 `FaaSLoRA` 同语义裁剪后的 `input_ids`

这样之后，客户端与服务端之间不再存在第二层 prompt 解释偏差。

## 2026-04-18 负载强度问题记录与最终实验分层

### 1. 问题结论

本轮重新分析已有结果、代码链路和同类论文设置后，确认之前 `500 requests / 500 adapters / time_scale=1` 的结果偏差，根因不是单纯的 TTFT/E2E 统计 bug，而是负载强度口径没有和论文主实验、压力实验分开。

`time_scale=1` 保留了真实 Azure trace 的到达时间尺度，因此它不是“假负载”。但在当前实验条件下，它同时叠加了：

- `100% LoRA` 请求；
- `500` 个候选 adapter；
- 长 prompt 和长输出；
- 本地 `4 x RTX 3090` 的有限 serving 能力；
- SGLang/ServerlessLLM 不具备 FaaSLoRA 的 adapter-aware scale-out 与 residency 策略。

因此该设置更像高强度压力测试。它可以用于证明 FaaSLoRA 在 burst 放大时更能抑制 queue/admission wait 和 adapter mismatch，但不应作为唯一主表负载。

### 2. 对论文实验的影响

FaaSLoRA 的研究动机仍然成立：现实中的 serverless LLM serving 确实会遇到多租户、突发请求、adapter 热点迁移、scale-out 和 adapter readiness 同时发生的问题。需要修正的是实验分层，而不是把该场景删掉。

后续论文应采用：

- 主实验：系统处于可服务但有压力的区域，展示常规比较。
- 压力实验：逐步压缩 trace 时间尺度，展示 overload sensitivity 和 tail latency 退化。
- 机制实验：拆分 `TTFT_e2e`、`TTFT_service`、`dispatch_admission_wait_ms`、LoRA I/O、cold start 和 scale-up affected requests。

### 3. 当前采用的推荐设置

快速 bring-up/debug：

- `500 requests`
- `500 adapters`
- `SLLM_TIME_SCALE_FACTOR=4.0`
- tag 建议包含 `_s4_e2ev3_v1`

正式主表候选：

- `4000 requests`
- `500 adapters`
- 先使用 `SLLM_TIME_SCALE_FACTOR=8.0`
- 若系统几乎无排队，再补 `4.0` 作为更高压主表候选

压力测试章节：

- `4000 requests`
- `500 adapters`
- `SLLM_TIME_SCALE_FACTOR in {8.0, 6.0, 4.0, 2.0, 1.0}`

### 4. 必须保持的公平性约束

无论主实验、消融还是压力实验，都必须保持：

- 同一 sanitized frozen pool；
- 同一 shared trace；
- 同一 adapter subset；
- 同一 tokenizer/prompt budget；
- 同一 `metric_schema_version=e2e_v3`；
- 同一 `avg/p50/p95/p99` 统计口径；
- 三系统结果均通过 audit 后才能比较。

当前 FaaSLoRA 和 baseline 链路已经统一到 `e2e_v3`，并保留：

- `TTFT_e2e`
- `E2E_e2e`
- `TTFT_service`
- `E2E_service`
- `dispatch_admission_wait_ms`

后续所有旧结果只能作为问题定位记录，不能直接进入最终论文主表。
