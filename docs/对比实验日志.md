# 对比实验日志

本文档用于记录与其他系统进行公平对比时的实验背景、源码核查、论文核查、日志排查、根因分析与结论。后续每新增一个 baseline，或对同一 baseline 进行新一轮关键排查，都应继续在本文档中追加记录。

## 记录原则

1. 所有正式结论必须同时参考三类证据：
   - 对方论文或官方文档
   - 对方真实源码或本地复现代码
   - 本地运行日志、结果 JSON 与共享 artifact
2. 不允许只依据某一次结果数值就下结论。
3. 不允许用估计值替代可真实观测的指标。
4. 如果某项指标在 baseline 中拿不到真实值，应明确记为 `null`，不能用近似值掩盖。
5. 对比实验中的数据输入必须一致：
   - 同一基座模型
   - 同一 shared trace artifact
   - 同一 shared LoRA subset artifact
   - 同一指标语义
6. 主实验背景始终围绕论文问题本身，不为 baseline 单独改题。

## 2026-04-15：ServerlessLLM 公平对比排查记录

### 1. 本轮背景

目标系统：

- 本系统：`FaaSLoRA`
- 对比系统：`ServerlessLLM`

当前正式问题背景：

- `many-LoRA concurrent serving`
- `serverless inference`
- 同一轮共享合成负载
- 所有请求都绑定 LoRA

当前重点模型：

- `Llama-2 7B`
- `Llama-2 13B`

### 2. 当前统一的公平性约束

当前正式对比中，以下约束已经固定：

1. 主场景必须是 `many-LoRA`，而不是普通单模型 serving。
2. 所有请求必须带 LoRA，即 `100% LoRA requests`。
3. 两个系统共用同一份 `trace artifact`。
4. 两个系统共用同一份 `LoRA subset artifact`。
5. 主指标统一为：
   - `TTFT_overall`
   - `TPOT`
   - `E2E`
   - `Tok/s`
   - `Cost/req`
   - `CE`
   - `SLO`

### 3. 为什么正式主实验必须使用 `100% LoRA requests`

结论：对当前论文主问题而言，`100% LoRA` 是最合理的正式主设置。

原因如下：

1. 当前论文的研究对象不是普通 LLM serving，而是 `many-LoRA` 并发 serving。
2. 如果混入无 LoRA 请求，会显著稀释：
   - LoRA 路由压力
   - LoRA 加载与缓存压力
   - GPU-ready 命中压力
   - 扩容后首次服务压力
3. 这会让对比结果偏向通用 serving，而弱化本论文真正的系统贡献。

补充结论：

- `LoRA/非 LoRA 混合流量` 可以作为后续补充实验
- 但不能取代当前正式主实验

### 4. 论文、源码与本地日志交叉结论

#### 4.1 ServerlessLLM 论文与当前本地场景并不等价

从 `ServerlessLoRA` 论文页面与相关预览信息可知，论文中对 `ServerlessLLM` 的对比场景明显比当前本地正式场景更轻：

- LoRA 数量更少
- many-LoRA 压力更低
- 硬件条件更强
- 并不是当前这种 `500 requests + 500 adapters + 100% LoRA requests` 的正式 many-LoRA 压测

因此，论文中 baseline 的绝对数值，不能直接当作本地绝对预期。

#### 4.2 ServerlessLLM 官方 many-LoRA 路径本身受限

本地官方文档与源码显示：

- LoRA 服务支持主要落在 `transformers backend`
- `vllm backend` 在当前 `Llama-2 + publicmix LoRA` 组合上不是可靠正式路径

本地 probe 现象包括：

- 7B 上出现 `all-zero token ids`
- 13B 上出现 `StreamInput must be either an integer or a list of integers`

因此，当前正式 baseline 策略是：

- 优先尝试 `vllm`
- 正确性 probe 失败后自动回退 `transformers`

这不是“照顾 baseline”，而是避免用错误路径得出无效结果。

#### 4.3 当前场景会真实放大 ServerlessLLM 的排队与协调问题

本地 `all-LoRA` 7B 正式 trace 统计为：

- 请求数：`500`
- 到达窗口：约 `64.5s`
- 平均到达率：约 `7.75 req/s`
- 平均输入：约 `1014.88 tokens`
- 平均输出：约 `150.79 tokens`
- 唯一 LoRA：`86`
- 无 LoRA 请求数：`0`

对应的历史异常结果显示：

- `TTFT` 极大
- `runtime_ttft` 却很小
- `serverless_overhead` 与 `queue_wait` 极大

这说明问题的主因不是 decode 慢，而是：

- 实例协调
- 排队等待
- serverless 路径开销

### 5. 关于“ServerlessLLM 为什么会特别差”的分层结论

#### 第一层：早期复现配置偏差

历史上我们曾经把 baseline 跑成：

- `min_instances=0`
- `keep_alive=0`

这会人为夸大冷启动和等待时间。该问题已修复。

另外，本地复现过程中还出现过一次“栈已就绪但外层检测脚本误判失败”的工程性问题：

- `start_serverlessllm_stack.sh` 在等待 `Ray runtime started.` 时硬依赖 `rg`
- 非交互环境下 `rg` 不在 PATH 中
- 导致 head 实际已经启动，但检测脚本持续报 `rg: 未找到命令`，最终超时报错

该问题已修复为：

- 优先使用 `rg`
- 若 `rg` 不可用，则自动回退 `grep -E`

因此，后续再出现“Ray 实际已启动但脚本判定失败”的概率已被显著降低。

#### 第二层：当前正式场景本身更严苛

当前本地正式场景，是围绕论文问题背景构造的 many-LoRA 并发场景，负载强度明显高于其论文中展示的 baseline 场景。

因此，如果 baseline 在这里暴露明显排队问题，这是合理现象，不应简单理解为“复现错误”。

#### 第三层：其系统结构在当前场景下的真实短板

从本地日志与 replay 结果看，`ServerlessLLM` 的瓶颈主要在：

- queue wait
- serverless overhead
- 实例路径协调

而不是：

- 模型纯 decode 性能
- 指标脚本计算错误

## 2026-04-18：Llama-2 7B 主表 workload 与压力 workload 分流决定

### 1. 本轮结论

在统一 `shared trace + shared adapter subset + e2e_v3` 口径下，`Llama-2 7B / 500 requests / 500 adapters / s4` 这一轮结果显示：

- `SGLang` 正常，且 offered load 与其稳定吞吐大致匹配。
- `ServerlessLLM-vLLM` 与 `FaaSLoRA` 都进入过载区。
- `FaaSLoRA` 这轮最主要的问题不在 `service path` 本身，而在 `dispatch/admission + scale-up readiness`。

因此，本轮不能直接作为主论文 headline 表结果继续滚到 13B/Qwen，而应先把 `Llama-2 7B` 收敛到“文献认可的主对比区间”。

### 2. 为什么要把主表 workload 与压力 workload 分开

结合 `ServerlessLLM`、`ServerlessLoRA`、`InstaInfer` 等论文的实验方式，本项目决定：

- 主表 workload：
  - 保持真实 Azure/ShareGPT 来源与 many-LoRA 语义
  - 但不使用会把所有 serverless 系统整体压穿的强度
  - 目标是让系统差异主要体现为 `readiness / routing / scale-out / adapter management`，而不是“谁先被压崩”
- 压力 workload：
  - 保留更重的长 prompt、更强 burst、更高请求数
  - 用于单独章节分析退化斜率、tail latency、扩容鲁棒性

### 3. 已落地到代码的调整

当前 clean-tree 已把 `Llama-2 7B` 的主论文主线与压力场景显式分开：

- `llama2_7b_auto500_main`
  - 用于主论文对比主表
  - 更快的扩容观测与反应窗口
  - 更大的 `max_num_batched_tokens`，使单卡 7B 的名义 8 路并发具备真正兑现的 batch-token 包络
- `llama2_7b_auto500_stress`
  - 保留更保守、更慢的扩容窗口
  - 用于后续压力测试章节

### 3.1 2026-04-19 追加：`faaslora_full` 场景层硬覆盖被修正

在 `s8_mainv1` 首轮复跑后，结果审计通过，但 `FaaSLoRA` 仍明显异常偏差。进一步对结果 JSON、配置文件与 runner 合并逻辑做交叉排查后确认：

- `llama2_7b_auto500_main` 已正确把
  - `arrival_window_s=2.0`
  - `scale_eval_interval_s=2.0`
  写入最终实验结果；
- 但 `faaslora_full` 场景层仍然额外写死了 `max_concurrent_loads=2`；
- 运行时 `sc_coord = {**coord_cfg, **sc.get("resource_coordination", {})}` 会让这个场景层值覆盖 workload profile 中原本希望使用的 `max_concurrent_loads=3`。

这意味着上一轮 `s8_mainv1` 的 FaaSLoRA 结果虽然在指标统计上是正确的，但并不是“主论文 main workload 按预期参数执行”的结果。

因此，当前已做两项修正：

1. 删除 `faaslora_full` 场景层对 `max_concurrent_loads` 的硬覆盖，让并行加载预算由 active workload profile 决定。
2. 在结果元数据中显式记录：
   - `max_concurrent_loads`
   - `warm_pool_size`
   - `scale_cooldown_s`
   - `scale_decision_interval`
   - `scale_up_alpha`
   - `scale_up_t_min`

后续如果再出现“workload 设了某参数，但运行结果没按这个值走”的情况，可以直接从结果 JSON 的元数据层定位，而不必再反向推断。

### 3.2 2026-04-19 追加：`s8_mainv1` 首轮结果的第一性原则归因

这一轮必须和更早的 `alllora_v5 / canonical_v1` 历史结果分开理解，不能只看 headline 数字。

#### 先看什么是“没坏”的

本轮 `s8_mainv1` 首轮结果已经通过统一审计：

- 同一份 `shared trace`
- 同一份 `adapter subset`
- 三系统 `500/500`
- `metric_schema_version = e2e_v3`
- `TTFT_e2e = dispatch_admission_wait + TTFT_service` 的分解关系成立

因此，这轮首先不是“指标脚本坏了”，也不是“shared artifact 不一致”。

#### 再看什么是“不能直接横比历史漂亮数字”的

更早的 `FaaSLoRA` 7B 结果中，确实存在很多 `1.2s ~ 1.5s` 量级的 `avg_ttft_ms`。但它们不能直接证明当前主链应该仍然是这个数量级，原因有两个：

1. 旧结果中大量样本尚未统一到当前严格的 `e2e_v3` 端到端主口径，很多只反映 admitted service-path，而没有完整暴露 `dispatch/admission wait`。
2. 不同轮次虽然 backbone/profile 名称相同，但必须区分：
   - 同一 trace 不同 time scale
   - 旧口径与新口径
   - 是否已经进入 shared-artifact 正式公平链路

因此，历史好看结果可以作为“机制曾经有效”的证据，但不能直接当作“当前正式主表应达到的数值”。

#### 本轮真正的第一根因

对 `s8_mainv1` 首轮结果做代码、结果 JSON、配置层和历史链路交叉核查后，当前确认的第一根因是：

- 主 workload `llama2_7b_auto500_main` 期望使用更快的 `max_concurrent_loads=3`
- 但 `faaslora_full` 场景层又把它覆盖回 `2`
- 在当前 live scale-up 逻辑里，`pressure_present` 时后台并行 scale-up 上限会退化为 `max_concurrent_loads - 1`
- 因此这一覆盖实际上把 4 卡机器的后台 scale-up 并行度压成了 `1`

这不是“参数偏一点”的小问题，而是会直接改变系统扩容形态的结构性覆盖错误：

- 系统名义上是 `4 × 3090`
- 实际首轮 `s8_mainv1` 却更接近“主实例 + 单后台串行冷启动”

于是后果按因果链依次放大为：

1. 第二、第三个 dedicated runtime 只能串行起来
2. backlog 在 cold-start 窗口内持续堆积
3. `dispatch_admission_wait_ms` 被首先拖爆
4. admitted 请求随后又在冷 runtime 内经历更长的 LoRA I/O 与 runtime queue
5. `TTFT_e2e` 和 `TTFT_service` 一起被放大

#### 这次我做的修正为什么是必要的，但还不能宣称“已经完全修好”

当前已做的修正包括：

1. 删除 `faaslora_full` 场景层对 `max_concurrent_loads` 的硬覆盖
2. 把 `max_concurrent_loads / warm_pool_size / scale_cooldown_s / scale_up_alpha / scale_up_t_min` 等关键协调参数写入结果元数据
3. 补充测试，保证 `faaslora_full` 以后不会再把 parallel load budget 硬钉死

这些修正是必要的，因为它们修的是“参数根本没按主 workload 生效”的根因，而不是在已有错误链路上继续调数字。

但它们还不是“全部根因都已消失”的证明。原因是：

- `s8_mainv1` 首轮里除了巨大的 `dispatch_admission_wait` 外，`TTFT_service` 自身也已经明显偏大
- 这说明除了 admission 前队列，admitted 之后的 cold runtime 路径也还有明显尾部
- 因而下一轮重跑的目标不是“验证数字一定立刻回到 1 秒级”，而是先验证：
  - dispatch/admission 是否显著回落
  - 扩容是否从串行变为更接近并行
  - FaaSLoRA 是否重新回到“至少明显优于 ServerlessLLM”的合理区间

#### 哪些东西当前还只是“风险点”，不能误判成第一根因

1. `max_model_len = 1024`
   - 当前三系统在 7B 正式 profile 下是统一的，不是单边不公平
   - 它主要影响的是 workload 代表性与服务后 token 分布解释，不是这次 `s8_mainv1` 首轮灾难性退化的第一根因
2. `scale_decision_interval`
   - 当前在 continuous-queue-v2 主线里，它并不是 live scale-up active control path 的主驱动参数
   - 因此不能把它误当作当前最该优先调整的热路径旋钮
3. 旧结果里更高的 `throughput_rps`
   - 跨不同 `time_scale_factor` 不能直接横比
   - 本轮 `s8_mainv1` 的 trace 时间跨度本来就更长，因此 throughput 只能在同一轮三系统之间比较，不能直接和更早 `s4 / canonical / alllora` 混比

### 4. 当前正式执行规则

1. 主表对比：
   - 优先使用 `llama2_7b_auto500_main`
   - shared trace 的 `time_scale_factor` 取“文献可接受、系统不过载”的区间
2. 压力测试：
   - 使用 `llama2_7b_auto500_stress`
   - 允许继续保留更重 `time_scale_factor` 或更高请求数
3. 只有当 `Llama-2 7B main` 先收敛，才继续滚后续 `13B / Qwen 7B / Qwen 14B`

因此，本轮不是 shared artifact 不一致、trace 错配或指标结构缺失问题。

- `ServerlessLLM-vLLM`

  - `TTFT_service_avg = 179390.80 ms`
  - `Dispatch_admission_wait_avg = 0.69 ms`

- `FaaSLoRA fix1`
  - `TTFT_e2e_avg = 538163.44 ms`
  - `TTFT_service_avg = 50552.53 ms`

这说明 `fix1` 相比前一轮已经显著改善，但系统仍然处于不可用于主表的异常状态。

#### 这轮为什么“开头看起来好，后面变差”

该现象不是随机波动，而是 open-loop trace 下的典型可解释规律：

1. replay 按固定 arrival schedule 放请求，不会等上一批完成；
2. live 日志早期只能看到已经完成的快请求；
3. 慢请求、冷启动请求、长队列请求会在后半段集中完成；
4. 因此 live 均值会随着 pending slow tail 回填而持续变差。

所以，不能用运行前几分钟的 live 平均值判断最终效果。最终必须以完整 JSON、audit 和 compare 为准。

#### `fix1` 的新根因：dedicated worker 的父进程 token 规划缺失

继续按第一性原则核对后发现，`fix1` 还有一个会影响 FaaSLoRA 的结构性问题：

- 主 `InferenceEngine` 有 `prepare_request()`，会在父进程侧完成 prompt guard、输出上限、token 预算规划；
- 但 dedicated scale-out worker 使用的是 `SubprocessInferenceEngineProxy`；
- 旧的 `SubprocessInferenceEngineProxy` 没有 `prepare_request()`；
- 因此 `ScenarioRunner._prepare_request_execution_plan()` 在父进程侧会退回到 raw trace token hint；
- worker 子进程真正执行时又会在 `InferenceEngine.generate()` 内部重新做 prompt guard。

这造成父进程和 worker 的视角不一致：

- worker 实际推理使用的是截断后的 prompt；
- 父进程的资源协调、KV 预算、成本统计、batch token 统计使用的是 raw trace hint；
- dedicated runtime 的 `input_tokens` 被显著放大；
- admission、resource coordination 和结果统计都被污染。

本轮三系统 token 统计对照能直接看到这个问题：

- `SGLang prompt_tokens_avg = 752.62`
- `ServerlessLLM prompt_tokens_avg = 735.02`
- `FaaSLoRA input_tokens_avg = 1013.29`
- `FaaSLoRA input_tokens_p95 = 1962`
- `FaaSLoRA input_tokens_max = 2047`

在同一 trace、同一 backbone、同一 prompt budget 下，FaaSLoRA 不应单边记录出明显更大的父进程侧 input token 分布。这不是论文机制优势或劣势，而是实验链路中的父子进程规划不一致。

#### 已完成的根因修复

当前已修复：

1. 为 `SubprocessInferenceEngineProxy` 增加 `prepare_request()`，在父进程侧复用与 worker 一致的 `InferenceEngine.prepare_request()` 逻辑。
2. 为 `SubprocessInferenceEngineProxy` 增加 `generate_prepared()`，使 dedicated worker 的执行路径和主引擎路径在调度层保持同一抽象。
3. 增加单元测试，验证 subprocess proxy 会在父进程统计前完成与 worker 一致的 prompt guard。
4. 修正测试 fake engine 的 `prepare_request()` 签名，使其与当前真实接口一致。

本地验证已通过：

- `python3 -m py_compile scripts/run_all_experiments.py tests/test_basic_smoke.py`
- `python3 -m unittest` 相关 4 个 targeted tests

#### 下一步判据

下一轮只需要重跑 `FaaSLoRA`，不需要重跑 `SGLang` 和 `ServerlessLLM`，因为 baseline 两个结果已经通过 audit，且 shared trace/subset 不变。

`fix2` 重跑后必须重点检查：

1. `FaaSLoRA input_tokens_avg` 是否回落到与 baseline 同一量级；
2. `Dispatch_admission_wait_avg` 是否明显下降；

#### `inst_1` 极端长尾的直接原因

进一步查看 `inst_1` 的最差请求，发现它们的 `lora_io_ms` 出现了 600000--750000 ms 级别数值。例如：

- `req_00202`
  - `lora_io_ms ≈ 716078 ms`
  - `admitted_service_ttft_ms ≈ 1918305 ms`
- `req_00048`

  - `admitted_service_ttft_ms ≈ 1810757 ms`
- `req_00050`
  - `lora_io_ms ≈ 752785 ms`
  - `admitted_service_ttft_ms ≈ 1875184 ms`

这不是正常的 NVMe/host-to-GPU LoRA 传输时间。真正原因是在线请求热路径里存在一个 `D1` 逻辑：

1. 请求先经过 `_resolve_lora()` 得到 LoRA 路径和 source tier；
2. 如果 source tier 是 `host/nvme`，系统会调用 `load_lora_to_gpu_and_measure()`；
3. 该函数为了“真实测量 LoRA 加载”，向 vLLM 发一个 1-token dummy generate；
4. 然后正式请求再执行一次真实 generate。

在低压或串行场景中，这个 dummy generate 可能看起来像“测量 LoRA load”。但在 open-loop、多请求、vLLM 已有队列的场景中，它会把运行时排队时间也计入 `lora_io_ms`，并且还额外制造一次真实 vLLM 请求。于是：

- `lora_io_ms` 不再是纯 adapter loading；
- service TTFT 被公式化地加上一个已包含排队的 dummy TTFT；
- 主实例在最早一批请求中被 dummy load/generate 放大阻塞；
- 后续 dispatch/admission wait 被进一步放大。

这解释了为什么 `fix2` 的 token 分布正确了，但延迟仍然崩坏：问题已经从“token 统计污染”推进到“在线热路径用 dummy generate 测 LoRA load，污染真实服务路径”。

#### 已完成的第二根因修复

当前已修复：

1. 在线 `_exec_request()` 路径不再调用 `load_lora_to_gpu_and_measure()` 发 dummy generate。
2. `TTFT_service` 改为从 request admission 到 first token 的墙钟时间：
   - `TTFT_service = (generate_start - admitted_time) + vLLM_TTFT`
3. `E2E_service` 改为从 request admission 到 response completion 的墙钟时间：
   - `E2E_service = response_done_time - admitted_time`
4. `lora_io_ms` 保留为诊断性来源层/预测性开销，不再用于拼接主 service latency。
5. 请求成功后仍会将 `host/nvme` 来源的 adapter 标记为 GPU resident，用于后续 affinity/residency 判断。

这个修复不改变 FaaSLoRA 的核心机制，也不改变 baseline、模型、数据或 LoRA pool。它修的是实验热路径中“用 dummy request 测量并污染真实 request”的根因。

## 2026-04-15：研究价值与基线选择补充结论

### 1. 为什么当前 many-LoRA serverless 问题仍然有研究价值

联网核查后，当前结论是：该问题不仅仍然存在，而且是当前大模型部署环境中的真实系统问题。

主要依据如下：

1. `ServerlessLLM`（OSDI 2024）表明，通用大模型的 serverless 推理仍然需要重点解决冷启动、checkpoint 加载和启动调度问题。
2. `Punica`（MLSys 2024）与 `S-LoRA`（MLSys 2024）表明，多 LoRA 并发 serving 已经成为独立且重要的系统问题，不能再被简单视为普通 LLM serving 的一个小变体。
3. 当前主流生产推理引擎和平台已经直接支持多 LoRA serving：
   - `vLLM` 官方文档支持按请求加载 LoRA，并支持运行时动态加载/卸载
   - `LoRAX` 官方文档明确支持按请求动态加载 LoRA，并可对多种不同 LoRA 做高吞吐批处理
   - `TGI` 文档也已经支持一次加载多个 LoRA adapter

这说明：

- 多 LoRA 服务不是“假设中的未来需求”
- 而是已经进入真实推理引擎与平台能力边界的当前问题

### 2. ServerlessLLM 的优点与边界

当前对 `ServerlessLLM` 的客观判断是：

#### 优点

1. 它是高质量、已发表的 serverless LLM 系统论文，研究问题明确。
2. 它的设计重心非常清晰：
   - checkpoint locality
   - 多层 checkpoint loading
   - live migration
   - startup-time optimized scheduling
3. 在“通用大模型 serverless 启动与迁移”这一问题上，它依然有代表性。

#### 边界

1. 它并不是围绕 `many-LoRA` 场景设计的。
2. 在当前本地复现路径上，它对 LoRA 的正式可用支持主要落在 `transformers backend`。
3. 对当前 `Llama-2 + publicmix LoRA` 组合，`vllm` 不是稳定正式路径。
4. 它在当前 `many-LoRA + 长 prompt + 高到达率` 场景下暴露出明显的队列与协调瓶颈。

因此：

- `ServerlessLLM` 适合作为“通用 serverless LLM baseline”
- 但不适合作为“many-LoRA 主基线”的唯一候选

### 3. ServerlessLLM 当前是否还能作为论文基线

可以，但定位必须调整。

当前更合理的定位是：

- `次要对比基线`
- `serverless 通用基线`
- `反映通用 serverless LLM 系统在 many-LoRA 场景下的不足`

而不是：

- 你的唯一主对比对象
- many-LoRA 问题上的最强专门 baseline

### 4. 当前更合理的对比对象选择

#### 第一层：主对比系统

这些系统与论文问题最接近，应优先考虑复现和对比：

1. `Punica`（MLSys 2024）
   - 关键词：`Multi-Tenant LoRA Serving`
   - 特点：明确面向多 LoRA 共享基座 serving
2. `S-LoRA`（MLSys 2024）
   - 关键词：`Serving Thousands of Concurrent LoRA Adapters`
   - 特点：明确面向海量并发 LoRA adapter serving

#### 第二层：次要系统

这些系统与论文问题部分重合，但不是 many-LoRA 专门系统：

1. `ServerlessLLM`（OSDI 2024）
   - serverless 通用基线
2. `MuxServe`（ICML 2024）
   - 多模型 serving / multiplexing 基线
3. `Aegaeon`（SOSP 2025）
   - 并发 LLM serving / GPU pooling 方向

2. `vLLM` / `LoRAX` / `TGI`
   - 更偏工程基线或生产系统参考
   - 不宜替代已发表研究系统

### 5. 论文叙事建议

当前更稳妥的论文叙事是：

1. 你的问题不是“通用 LLM serving”
2. 你的问题是“many-LoRA serverless inference”

因此，本轮“失败”更准确地说是：

- baseline 在当前负载下发生严重排队
- replay 客户端在 `600s` 读超时前仍未等到响应
- 于是被记录为失败

### 6. 关于“服务失败问题”是否应该修

结论：应该修，但修复目标必须定义清楚。

当前最合理的修复目标不是：

- 去给 `ServerlessLLM` 增加新的 LoRA-aware 系统机制
- 或为了让它变好而改变题目、数据、LoRA 压力

而是：

- 去掉 replay 侧 `600s` 读超时对正式结果的截断
- 让 baseline 把“真实但很慢”的结果完整跑出来

这是公平的，因为：

1. 它不改变模型、不改变 shared trace、不改变 shared LoRA subset。
2. 它不改变 baseline 的系统设计，只改变我们对其完成情况的观测窗口。
3. 它不会影响 `FaaSLoRA`、基座模型、数据集和主实验链路。

因此，本轮后续采取的 baseline-only 修复是：

- 将 `run_serverlessllm_fair_experiment.sh` 的默认 replay 超时从 `600s` 提高到 `3600s`

该修复的目标不是“让 ServerlessLLM 变快”，而是：

- 避免还在队列中的请求被客户端过早判死
- 让正式对比先从“观测被截断”变成“真实结果完整返回”

### 7. 当前阶段性判断

1. `FaaSLoRA` 在当前 many-LoRA 问题背景下，已经表现出明显系统性优势。
2. `ServerlessLLM` 的当前主要问题并不是 decode 性能，而是 queue / coordination / serverless overhead。
3. `ColdStart` 仍然是 `ServerlessLLM` 的真实强项，这部分值得在论文中客观呈现。
4. `ServerlessLLM` 仍可保留为“通用 serverless LLM 基线”，但不应成为唯一主基线。
5. 如果后续继续补 baseline，优先级更高的应是：
   - `Punica`
   - `S-LoRA`

### 4. 结合论文与代码实现的根因分析

#### 4.1 论文目标与当前问题背景并不一致

`ServerlessLLM` 论文的核心贡献是：

1. 多层 checkpoint 加载
2. 运行时 live migration
3. 启动时间优化的 model scheduling

其设计中心是通用大模型 serverless inference 中的：

- checkpoint locality
- model startup latency
- instance migration

而不是当前论文的 many-LoRA serverless inference 问题。

这与当前结果是吻合的：  
它的 `ColdStart` 仍然相对较好，但 `TTFT` 主体却被 many-LoRA 条件下的 queue/coordination 拖垮。

#### 4.2 当前代码路径不是 LoRA-aware 路由

本地 `RoundRobinRouter` 实现显示：

- 请求首先进入全局 `asyncio.Queue`
- load balancer 按 `round-robin` 方式选实例
- 选实例时只看 queue capacity，不看 adapter 命中或 LoRA ready 状态

也就是说，这条正式路径并不做：

- adapter affinity routing
- hot-adapter aware placement
- LoRA-aware warmup

这与 `FaaSLoRA` 的系统目标根本不同。

#### 4.3 `transformers` LoRA 路径是“请求分配后再加载”

当前正式可用路径下，`ServerlessLLM` 会在请求已经分配到实例之后，再执行：

- `load_lora_adapter(...)`

如果该实例上尚未加载对应 adapter，则会发生真实加载。  
这意味着：

1. LoRA ready 不是路由前状态，而是请求落到实例后的后处理。
2. 请求即使已进入 ready instance，也可能因为 target adapter 未就绪而继续等待。

这正是 many-LoRA 场景中的关键不匹配来源。

#### 4.4 当前主瓶颈不是 decode，也不是冷启动

新结果中：

- `Runtime_TTFT_avg_ms ≈ 73.15ms`
- `Cold_start_avg_ms ≈ 7767.91ms`
- `TTFT_overall_avg_ms ≈ 715709.07ms`
- `avg_queue_wait_ms ≈ 715137.64ms`

这说明：

- decode 不是主要瓶颈
- cold start 不是主要瓶颈
- 主要瓶颈就是 queue wait 与 serverless overhead

因此，当前 baseline 的核心问题不是“模型慢”，而是“在当前 many-LoRA 负载下，它的请求组织方式和 adapter-ready 路径不适合这个问题”。

#### 4.5 在当前公平约束下，参数调节空间已经接近耗尽

当前正式 deploy 已经满足：

- `min_instances=1`
- `max_instances=4`
- `keep_alive=10`
- 4 个实例都真实拉起并承接了请求

这说明问题已经不是：

- 没扩出来
- keep_alive 太激进
- min/max instance 配错

而是在 4 张 GPU 都已经被用满的情况下，系统仍然无法把 many-LoRA 请求组织成低等待服务。

### 5. 是否还能在我们的原则下把它拉回“可比范围”

结论：基本不能，至少不能通过“只调官方参数、不改系统设计”的方式做到。

原因如下：

1. 目前已经证明，扩大观测窗口之后它可以 `500/500` 完成服务。
2. 但主瓶颈来自系统设计与问题背景的不匹配，而不是单一脚本 bug。
3. 如果继续想把它拉回可比范围，只剩下两条路：
   - 降低 LoRA 压力、降低到达率、换更强硬件
   - 给它加入 LoRA-aware 路由、adapter 预热或更强 batching
4. 前者会改变题目和正式场景，不符合公平原则。
5. 后者会改动 baseline 核心设计，不再是 `ServerlessLLM`。

因此，在当前原则下，`ServerlessLLM` 最多只能被修到：

- “能完整服务”

但很难被修到：

- “在当前正式 many-LoRA 场景下进入与你系统相近的性能区间”

### 6. 当前定位建议

基于这轮结果，当前对 `ServerlessLLM` 的定位应当进一步收紧为：

- 它可以保留为“通用 serverless LLM 基线”
- 它不适合充当 many-LoRA 问题的唯一主基线
- 在本文中，它更适合回答：
  - 通用 serverless LLM 系统在 many-LoRA 场景下会暴露什么问题

而真正更匹配的问题基线，应优先考虑：

- `Punica`
- `S-LoRA`

### 2. 当前新根因

`fix3` 的异常集中在 FaaSLoRA 的单个 vLLM runtime：

| Instance | Requests | avg runtime TTFT ms | avg service TTFT ms | avg dispatch wait ms | avg LoRA I/O ms |

| inst_3 | 231 | 20713.1 | 22634.0 | 662971.9 | 12.0 |
| inst_4 | 24 | 55755.1 | 57623.3 | 96267.8 | 15.2 |

其中 `inst_1` 在早期请求中出现 `vllm_ttft_ms` 级联膨胀，最高超过 1900 s。该现象符合以下规律：

- 刚开始运行时指标较好；
- 随着进入同一 runtime 的不同 LoRA adapter 数量增加，服务路径 TTFT 快速恶化；
- LoRA I/O 没有同步膨胀，说明瓶颈不是 artifact load；
- 其他实例启动后能承接大部分请求，但早期被压入 `inst_1` 的请求已经造成极长 runtime 队列。

第一性原则判断：FaaSLoRA 之前只按 `runtime_concurrency_cap / max_num_seqs` 控制并发请求数，却没有把 vLLM 的 `max_loras` 当成并发物理资源。当前模型配置中 `max_loras=8`，但请求路由可在同一 runtime 内并发引入超过 8 个不同 active LoRA。vLLM 的 `max_loras` 不是简单缓存大小，而是单个 engine 在并发执行中可承载的 LoRA adapter 多样性上限。超过这个上限后，请求会在 runtime 内部排队或退化，表现为 `vllm_ttft_ms` 极端膨胀。

这解释了“刚开始好、过一会变差”的反复规律：早期 active LoRA set 尚未填满；当突发流量把多个不同 adapter 同时压入同一 runtime，系统进入 active-LoRA saturation，随后 TTFT 被 runtime 内部队列拖垮。

### 3. 已实施的根因修复

本轮修复只改 FaaSLoRA 控制面，不改基座模型、LoRA 工件池、baseline 环境或数据集：

- `InstanceSlot` 增加 `active_adapter_counts`，记录当前 runtime 正在执行的 LoRA adapter 多样性；
- `Router` 增加 `max_active_loras`，路由时优先避开 active LoRA set 已满的 runtime；
- `_exec_request` 在请求进入 runtime 前增加硬门控：
  - active request lane 未超过 `runtime_concurrency_cap`；
  - 若为 LoRA 请求，active LoRA set 未超过 `max_loras`，或该 adapter 已在 active set 中；
- 因该等待发生在请求真正进入 runtime 之前，等待时间计入 `dispatch_admission_wait_ms`，不计入 `TTFT_service`；
- 请求完成后释放 active LoRA 引用，并唤醒等待中的 admission。

这次修复的预期影响：

- `inst_1` 不应再出现超过 `max_loras` 的 active adapter 多样性；
- `TTFT_service` 和 `vllm_ttft_ms` 应显著下降；
- `dispatch_admission_wait_ms` 可能上升或重新分布，因为原本错误进入 runtime 内部的排队被前移到 FaaSLoRA admission；
- 如果总负载仍超过系统可服务能力，`TTFT_e2e` 仍会较高，但其组成应从“不可控 runtime 崩坏”变成“可解释 admission/scale-out 排队”。

## 2026-04-19 Llama-2 7B `fix5` 前置修复：primary runtime 与 scale-out runtime 拓扑一致性

### 1. 继续复盘 `fix4` 日志后的新增判断

进一步读取 `fix4` 的连续日志后，故障链条可以拆成两层：

- 第一层根因：`inst_1` 的 vLLM background loop 在约 34 分钟处超时；
- 第二层连锁故障：`inst_1` 被移除后，系统尝试在 GPU0 上补新实例，随后因为 GPU0 仍被失败 runtime / 主进程残留占用而 OOM，再触发 foreign GPU consumer 保护。

上一节已经修复第二层：失败 runtime GPU quarantine、无空闲 GPU 时不 round-robin 复用、worker 启动失败强制 SIGKILL、异常路径清理 instance pool。

本次新增分析聚焦第一层：为什么最先崩的是 `inst_1`。

日志规律如下：

- `inst_1` 长时间保持 `active=8`；
- GPU0 常驻约 16.8GB；
- `inst_1` 的 GPU cache 很小，`cache[g/h/n]` 约为 `1/80/85`，说明问题不是 GPU adapter 驻留过多；
- 其他 scale-out 实例也在处理请求，但它们是 dedicated subprocess；
- `inst_1` 是 primary runtime，仍运行在主控制进程内。

第一性原则判断：`fix3/fix4` 已经把 active LoRA 多样性约束到 `max_loras`，因此 `inst_1` 再次崩溃不应继续归因于 LoRA active set 超限。更合理的根因是 runtime 拓扑不一致：scale-out runtime 通过子进程隔离 vLLM async engine，而 primary runtime 仍和调度、live progress、GPU 检查、warmup、autoscaler 等控制面共用同一个 Python 事件循环和进程资源。这会使 primary 在长时间高并发下比 scale-out runtime 更容易出现 vLLM background loop timeout。

这也解释了之前反复出现的经验规律：

- 刚开始运行时效果较好，因为 primary 尚未长期承压；
- 运行一段时间后，控制面任务、监控输出、admission 等逻辑与 primary vLLM loop 叠加，primary 逐渐成为隐藏瓶颈；
- 后续 scale-out runtime 本身并未同样崩溃，说明问题不是所有 vLLM 实例天然不稳定，而是 primary 的进程拓扑不公平。

### 2. 已实施的结构性修复

本轮修复只改变 FaaSLoRA 实验运行时拓扑，不改基座模型、LoRA 工件池、共享 trace、adapter subset、baseline 代码或指标口径：

- 当被选中的 serving 场景是 vLLM `auto/dedicated` 模式时，不再提前在主进程初始化 primary vLLM engine；
- 场景开始时，primary runtime 使用 `SubprocessInferenceEngineProxy.spawn()` 启动，和 scale-out runtime 走完全相同的子进程隔离路径；
- `SubprocessInferenceEngineProxy` 记录该 runtime 的物理 `runtime_gpu_ids`，后续 reinitialize 仍使用原物理 GPU 组，避免把子进程本地 `visible_device_ids=[0]` 误当作宿主机 GPU0；

这说明 baseline replay 仍存在 service-boundary 不闭合问题，不能把本轮 baseline summary 作为最终论文结果。

### 3. 根因分析

SGLang 与 ServerlessLLM 都通过 HTTP replay 进入系统，但二者暴露的时间戳不同：

- SGLang 提供 `request_received_ts`、`response_sent_to_client_ts`、`request_finished_ts`；
- ServerlessLLM-vLLM 额外提供 `backend_started_at`、`first_token_at`、`last_token_at`、`finished_at`、`queue_wait_ms`、`runtime_ttft_ms`。

旧 baseline replay 的错误是：

1. `service_ttft_ms` 优先使用服务端 first-token 时间；
2. `overall_ttft_ms` 却仍优先使用客户端首个 HTTP chunk；
3. 对 ServerlessLLM 非流式 JSON 响应来说，客户端首 chunk 实际接近完整响应，而不是首 token；
4. 对暴露 `backend_started_at` 的 ServerlessLLM 来说，`request_received_at -> backend_started_at` 是后端 queue/admission wait，不应该算进 `TTFT_service`。

因此旧结果把不同时间轴混在一起：ServerlessLLM 的 `TTFT_service` 被 queue wait 放大，而 `overall_ttft` 又被非流式响应 chunk 放大，导致请求级公式不闭合。

### 4. 修复原则

按 FaaSLoRA 的语义统一三系统：

- `dispatch_admission_wait_ms = scheduled arrival -> admitted/backend-started`
- `TTFT_service = admitted/backend-started -> first generated token`
- `E2E_service = admitted/backend-started -> response finished`
- `TTFT_e2e = dispatch_admission_wait_ms + TTFT_service`
- `E2E_e2e = dispatch_admission_wait_ms + E2E_service`
- `TPOT = (E2E_service - TTFT_service) / max(output_tokens - 1, 1)`

其中，对于 ServerlessLLM：

- `request_received_at -> backend_started_at` 归入 `dispatch_admission_wait_ms`
- `backend_started_at -> first_token_at` 归入 `TTFT_service`
- `backend_started_at -> finished_at` 归入 `E2E_service`

这不是为了让哪个系统好看，而是为了把“排队等到可执行 runtime”和“runtime 已经开始服务后的 prefill/decode”分开，和 FaaSLoRA 的 admission/service 边界一致。

### 5. 旧 raw replay 上的修复后预估

在不重跑、仅用旧 raw replay 时间戳重算时：

- SGLang 变化很小：`TTFT_e2e_avg` 从约 `18614.7 ms` 修正为约 `18601.3 ms`，因为它本身是流式且 queue 边界简单；

本轮不能直接进入最终论文主表，因为 baseline replay 代码已修复但需要重新跑生成新的 raw replay/summary。可以保留的分析判断是：

## 2026-04-19 Llama-2 7B s8_mainv3：timeout 口径与 FaaSLoRA 越跑越差根因

### 1. 论文口径核对

ServerlessLLM 在 OSDI'24 评测中报告了 `300-second timeout` 下的 fulfilled request ratio：例如 OPT-30B 上 ServerlessLLM 可在 300s 内完成 89% 请求，而 Ray Serve with Cache 只有 26%。因此，`completion_rate@300s` 或 `effective service rate@300s` 可以作为 serverless LLM 场景中的合理 deadline 指标。

但 300s timeout 不应替代完整延迟指标。它的正确用途是补充回答：“在一个明确 deadline 内，系统实际服务了多少请求和 token”。主表仍应保留 `TTFT_e2e`、`E2E_e2e`、`TPOT`、`SLO attainment`、`cost`；deadline 表用于避免只看均值时掩盖超长尾。

- SGLang：`E2E<=300s = 500/500`，`CompletionRate = 1.0000`；

因此，timeout 指标不会“拯救”当前 FaaSLoRA 结果；它反而更清晰地证明当前瓶颈确实存在，必须修调度根因。

### 3. FaaSLoRA 越跑越差的根因

旧结果显示 FaaSLoRA 的 `TTFT_service_avg ≈ 1341 ms`，`cache_hit_rate = 100%`，`service_overhead ≈ 13 ms`。这说明 LoRA residency、adapter readiness、加载协调本身没有失效；真正拖垮 `TTFT_e2e` 的是 `dispatch_admission_wait_avg ≈ 1267 s`。

代码根因在 `Router._routing_key`：旧排序把 active runtime 是否已满放在较后位置，adapter affinity、历史 service cost、handoff reservation 可能压过 runtime capacity。随后 `_exec_request` 如果选中的 slot 不满足 `_slot_can_accept_runtime_request`，会等待条件变量，而不是立即换到别的可接收 slot。结果就是：系统越运行，adapter-affinity 信息越丰富，越可能把请求吸到“看起来最优但当前满载”的实例，形成长排队。

### 4. 根因修复

已在 `serverless_llm_experiment_retry14_baseline/faaslora/experiment/instance_pool.py` 中把 runtime capacity 提升为硬优先级：

- 新增 `_runtime_capacity_penalty(slot)`；
- 当 `active_requests >= runtime_concurrency_cap` 时，该 slot 在路由排序中先于 affinity/cost 被惩罚；
- `_routing_key` 排序顺序调整为：runtime capacity -> active LoRA capacity -> handoff reservation -> active request count -> service/cost；
- 最小单测验证：当 `inst_1.active_requests = runtime_cap` 且 `inst_2.active_requests = 0` 时，router 选择 `inst_2`。

该修复不改变基座模型、不改变 LoRA pool、不改变 shared trace、不改变 baseline、不改变指标口径，只修复 FaaSLoRA 自身调度中“选中不可接收实例后等待”的根因。

- baseline 在请求真正进入计时窗口之前，就已经完成：
  - `chat -> prompt` 渲染
  - prompt guard / token budget 裁剪
  - tokenizer encode
- FaaSLoRA 之前却把 `_prepare_request_execution_plan(...)` 放在 `_exec_request()` 热路径里，而且是在 request 已经通过 dispatch/admission 之后。

因此，FaaSLoRA 会把一部分本来不该属于“服务路径”的 CPU tokenizer / prompt 准备时间，直接算进：

- `TTFT_service`
- `pre-runtime service shell`
- 进一步算进 `TTFT_e2e`

- `TTFT_service_avg` 一直在 `~1.1s`
- 其中 `runtime_ttft_avg` 只有 `~0.42s`
- 两者之间剩下的那段壳子时间，不可能主要由 `avg_lora_io_ms ≈ 16.8ms` 解释

所以，下一轮最有依据的修复不是再碰 dispatch，而是：

- 在 replay 开始前预先为每个 request 计算并缓存 `RequestExecutionPlan`
- `_exec_request()` 只消费已经准备好的 `prompt/input_tokens/max_tokens`

### 4. 首 token 前分段时间轴

当前系统里，首 token 前可以拆成：

```text
scheduled arrival
  -> arrival_release_lateness
  -> dispatch_window_wait
  -> runtime_slot_wait
  -> pre_runtime_service_shell
  -> runtime_ttft
  -> first token
```

对应关系：

```text
DispatchWait
  = arrival_release_lateness
  + dispatch_window_wait
  + runtime_slot_wait

TTFT_service
  = pre_runtime_service_shell
  + runtime_ttft

TTFT_e2e
  = DispatchWait + TTFT_service
```

### 5. 哪个指标最能支撑“LoRA 预加载”

目前最能直接支撑 LoRA 预加载/adapter readiness 的，不是 `runtime_ttft`，而是：

1. `scaleup_first_service_ttft`
2. `TTFT_service`
3. `avg_lora_io_ms`

原因是：

- `runtime_ttft` 起点太晚，更偏 backend prefill/decode；
- `TTFT_service` 才真正覆盖了 adapter resolve/load 到首 token 的整段服务路径；
- `scaleup_first_service_ttft` 最接近“新函数 ready 后首批请求多久看到首 token”，更适合证明 LoRA 预加载的收益。

当前实验现状是：

- LoRA 预加载机制已经开始在 `scaleup_first_service_ttft` 上体现优势；
- 但 `TTFT_service` 还混入了过重的非 LoRA service shell；
- 因此下一轮修复应优先把 request-plan 准备前移出热路径，再看 `TTFT_service / TTFT_e2e / CE` 是否同步改善。

本轮新增的诊断拆分显示：

1. `RPC channel acquire` 基本不是瓶颈。
2. `LoRA I/O` 也不是当前最该打的点。
3. 当前 `parent_rpc_overhead` 几乎就是 `parent response pickup delay`。
4. 因此下一轮真正值得验证的，不再是 dispatch，而是：
   - parent 侧如何更快读到 worker 已经准备好的响应
   - 也就是 `subprocess RPC / response pickup` 这条链

### 4. 它和“LoRA 预加载贡献”是什么关系

这轮结果并没有削弱 LoRA 预加载的意义，反而帮我们把因果链理顺了：

- `avg_lora_io_ms` 很小，说明预加载确实已经把远程 LoRA 传输代价压得很低；
- `scaleup_first_service_ttft` 继续是最适合展示 LoRA 预加载收益的机制指标；
- 但整体 `TTFT_service` 仍被更重的 service shell 盖住，所以 headline 还没完全体现出 LoRA 机制优势。

也就是说：

- 现在不是“LoRA 预加载没有效果”；
- 而是“它的效果已经存在，但仍被 parent-side service shell 更大的成本压住”。

### 5. 下一轮修复原则

下一轮不再继续围着 `dispatch gate` 打转，而是：

1. 保留已经验证有效的 `request-plan precompute`
2. 保留 `strict gate + notify(1)` 这条稳定 dispatch 状态
3. 直接切到 `parent_rpc_response_pickup_delay`

实现目标是：

- 把 parent 侧等待 worker 响应的路径从主 event loop 的拥塞中剥开；
- 让 `parent_rpc_overhead` 的主要组成部分真正下降；
- 进一步带动 `TTFT_service / E2E / CE / InfraCE`

如果这一刀有效，那么后续 headline 指标改善才会更接近你的真正系统贡献，而不是继续被外层服务壳子盖住。

3. 如果新口径下 `parent_rpc_overhead_ms` 仍大，而 `pickup/acquire` 很小，再继续处理 parent-side RPC transport 或 generate 外壳。

这一步符合我们现在的修复原则：先把因果链拆干净，再做最大收益修复，避免继续局部打补丁。

### 2. 根因是什么

继续读代码后，根因落在 `ExperimentStack.sync_local_tier_paths()` 的调用边界：

- `resolve_lora()` 每个请求都会调用它；

所以这不是 LoRA 预加载无效，而是预加载后的 fast path 被每请求全量同步文件系统/registry 扫描盖住。

### 3. 本轮修复

修复方式：

- 给 `sync_local_tier_paths()` 增加 TTL 缓存；
- 默认热路径 `force=False`，短时间内不重复全量扫描；
- preload 完成、HOST promotion 完成等真实 tier 变更处使用 `force=True`；
- 保留 `_host_paths / _nvme_paths` 的语义，不用静默 fallback，也不删除预加载机制。

这条修复对齐我们的主线贡献：

- LoRA 预加载已经把 `lora_io_ms` 压低；
- 现在要移除的是预加载命中后不该存在的控制面扫描；
- 如果修复有效，`TTFT_service` 应更接近 `runtime_ttft + LoRA_IO`，论文里的“预加载降低首 token 前准备时间”会更有说服力。

### 1. 本轮结果

`lifecyclefix1` 确认了 service-path 修复稳定存在：

- `TTFT_avg_ms: 655.2 -> 586.5`
- `TTFT_P95_ms: 673.6 -> 727.6`
- `TTFT_P99_ms: 10096.9 -> 6992.0`
- `TTFT_service_avg_ms: 421.1 -> 427.2`
- `Pre_runtime_service_shell_avg_ms: 6.2 -> 6.6`
- `Dispatch_admission_wait_avg_ms: 234.1 -> 159.3`
- `SLO_attainment: 0.978 -> 0.986`

这说明短 TTL 并没有破坏推理热路径，早期尾部还略有改善。

但成本链收益不足：

- `infra_gpu_seconds_total: 5907.3 -> 5783.4`
- `infra_cost_per_request_usd: 0.0098455 -> 0.0096389`
- `InfraCE: 14.206 -> 14.188`

也就是说，虽然 scale-down 事件从 1 次变成 6 次，但 GPU 秒只降约 2.1%，没有形成论文需要的 serverless 成本优势。

### 2. 为什么收益不足

进一步用 request timeline 拆开后发现：

- 单个实例存在大量 20~50s 空闲窗口；
- 但全系统合并后的 idle gap 最大只有约 14.7s；
- 当前 live scale-down 逻辑要求 `backlog=0 && active_requests=0 && busy_ratio=0`，也就是“全系统空闲”后才允许回收；
- 因此一个 runtime 明明已经空闲很久，只要其他 runtime 还在生成，它也不能被回收。

这解释了 `lifecyclefix1` 的现象：

- `inst_3` 从 offset 85.8s 活到 1401.8s；
- `inst_5` 从 offset 304.8s 活到 1547.0s；
- 这些实例中途有长空闲，但被全局 pressure 门控挡住。

### 3. 本轮之后的根因修复

下一刀从“全局空闲回收”改成“实例级 idle TTL 回收”：

- `InstanceSlot` 增加 `last_idle_at`；
- 请求释放 runtime slot 后，如果该 slot `active_requests == 0`，更新 `last_idle_at`；
- scale-down candidate 必须满足：
  - 非 primary instance；
  - `active_requests == 0`；
  - `load_queue_depth == 0`；
  - `runtime_forwarding_active == 0`；
  - `now - last_idle_at >= scale_down_duration_s`；
- 即使全系统仍有部分 active/backlog，只要移除该 idle slot 后剩余 runtime 总容量仍能覆盖当前可见 work，也允许回收；
- 如果 autoscaler 正在要求 scale-up，则不做 idle 回收，避免扩缩同时抖动。

预期影响：

1. `infra_gpu_seconds_total` 应明显低于 `lifecyclefix1`。
2. `InfraCE` 应明显上升。
3. `TTFT_service / PreRuntime / RuntimeTTFT` 应基本不变。
4. 如果 `TTFT_P95/P99` 明显恶化，说明实例级 TTL 回收太激进，下一步应增加预测式提前 scale-up 或短 TTL warm standby，而不是回退到全局 idle。

### 2. 根因

- GPU 秒下降一部分；
- tail TTFT / DispatchWait / E2E 被下一次冷启动反噬；
- `InfraCE` 有改善但不稳定，不能成为主论文结论。

因此下一刀不能继续调小 TTL，也不能回退到永不 scale-down，而应该让实例级 scale-down 使用 cold-start break-even：只有预计 idle window 已经覆盖一次重启成本时，才允许释放该 runtime。

### 3. 本轮之后的修复方向

新增 `scale_down_startup_break_even_s` 和 `scale_down_break_even_factor`：

- 有 observed scale-up cold-start 时，使用最近样本估计 restart cost；
- 没有 observed 样本时，使用配置 floor；
- scale-down candidate 的最小 idle 时间取 `max(scale_down_duration_s, restart_cost * factor)`；
- global low-load scale-down 和 under-pressure idle-slot scale-down 都走同一 break-even 门。

预期影响：

- `scale_up_events / scale_down_events` 应明显低于 `antithrash1` 的 `21/21`；
- `TTFT_P95/P99` 和 `Dispatch_admission_wait` 应下降；
- `infra_gpu_seconds_total` 可能高于 `antithrash1`，但应低于纯常驻/过保守方案；
- `InfraCE` 的目标不是单纯靠少算 GPU 秒，而是在 `E2E` 不明显恶化的前提下体现 serverless 成本优势。

### 4. Baseline 成本口径同步

SGLang 作为 serverful baseline，runtime 在 replay 前已经启动并占用 GPU。为了让 `InfraCE` 口径公平，SGLang summary 现在纳入 launch-server 到 `/v1/models` ready 的静态启动 GPU 秒；这只改变基础设施成本统计，不改变 SGLang 的请求路径、调度能力、TPOT 或 TTFT。

## 2026-04-21 深挖：baseline token 预算不一致与 FaaSLoRA 自适应生命周期

### 1. 为什么旧 baseline 结果不能直接作为最终公平结论

这轮重新审计 SGLang 和 ServerlessLLM 后发现，旧结果里三系统实际服务的 completion token 分布不一致：

- FaaSLoRA completion token 最大值受模型 profile 的安全预算约束；
- SGLang 虽然经过 prompt guard，但 replay guard 之前会在 decode 后再次 re-encode，从而可能比 FaaSLoRA 更激进地缩短 prompt，让部分请求获得更大的输出预算；
- ServerlessLLM wrapper 没有把同一套 prompt guard 传给 replay，因此旧结果中有大量请求输出超过当前主线预算。

这会同时影响 `E2E`、`TPOT`、`Tok/s`、token cost、`CE` 和 `InfraCE`。因此旧结果只能用于定位问题，不能作为论文最终公平对比。

### 2. 本轮修复

已完成的公平口径修复：

- `replay_openai_trace.py` 的 prompt guard 改为与 FaaSLoRA 一致：只按 token id 截断一次，不在 decode 后继续压缩 prompt；
- ServerlessLLM 复现脚本也传入同一套 prompt guard；
- baseline summary 记录 `avg/max prompt tokens`、`avg/p95/p99/max completion tokens` 和 `metrics_source_counts`；
- compare 脚本新增 served-token 分布审计表；
- SGLang summary 纳入 launch-to-ready 静态 GPU 启动时间；
- FaaSLoRA cost 使用 vLLM 暴露的实际 prompt token 数；
- FaaSLoRA `scale_down_duration_s` 和 `scale_down_cooldown_s` 支持 `auto`，由 trace 间隔 + observed/predicted restart cost 推导，避免为 Llama-2 7B 写死秒数。

### 3. 下一轮判断顺序

下一轮结果必须按下面顺序看：

1. 先看 token audit，确认 `OutTok>256` 对三系统都为 0；
2. 再看 SGLang `InfraStartupGPU_s` 是否非 0；
3. 再看 FaaSLoRA `scale_up/down_events`、`infra_gpu_seconds_total`、`TTFT_service` 和 `DispatchWait`；
4. 最后才比较 `E2E`、`Cost/req`、`CE`、`InfraCE`。

如果 token audit 仍不一致，不允许解读 CE 或论文结论。

- queue threshold 改为按当前 runtime capacity 自适应，而不是固定 backlog 阈值

因此这轮可以确认：FaaSLoRA 的首 token 前控制面已经恢复到比 `tiersyncfix1/lifecyclefix1` 更好的 dispatch 等待区间。

### 3. 和 baselines 的公平审计

`dispatchaware1` 的 token audit 与 baseline 对齐：

- FaaSLoRA: `PromptTokAvg=769.92`, `OutTokAvg=104.49`, `OutTokMax=256`, `OutTok>256=0`
- SGLang: `PromptTokAvg=768.96`, `OutTokAvg=104.89`, `OutTokMax=256`, `OutTok>256=0`
- ServerlessLLM: `PromptTokAvg=768.81`, `OutTokAvg=102.32`, `OutTokMax=256`, `OutTok>256=0`

执行包络也对齐：`MaxModelLen=1024`, `RuntimeCap=2`, `MaxSeqs=2`, `MaxLoras=4`。

但新增 topology audit 暴露出下一层公平性/解释性差异：

- FaaSLoRA: `TP=1`, `GPU/Req=1`, `Topology=single_gpu`
- SGLang: `TP=4`, `GPU/Req=4`, `Topology=model_parallel_tp4`
- ServerlessLLM: `TP=1`, `GPU/Req=1`, `Topology=single_gpu`

这解释了为什么当前 FaaSLoRA 的 `TTFT_e2e` 已经接近 SGLang，但 `TPOT/E2E/CE` 仍明显输给 SGLang：SGLang 当前不是 4 个单 GPU serverful replica，而是一个 TP4 model-parallel server，每个请求可使用 4 张卡降低 per-request decode latency。

### 4. 下一步根因链

当前不应继续围绕 dispatch 或 TTFT_service 打补丁。下一层必须拆清：

1. 论文主对比中，SGLang 是否应保留 `TP=4` 作为 serverful upper bound，还是应补一个 `4 x TP1` 的 serverful data-parallel baseline；
2. FaaSLoRA 的贡献是 serverless LoRA lifecycle / dispatch / preloading，不应被“baseline 每请求使用 4 GPU model parallel，而 FaaSLoRA 每请求使用 1 GPU scale-out”的拓扑差异淹没；
3. compare 脚本已新增 topology audit，以后所有主表解读必须同时报告 `TP / GPUPerRequest / RuntimeGPUs / Topology`；
4. 如果目标是主指标 CE/InfraCE 超过 SGLang，需要么降低 FaaSLoRA 的 infra cost 一个数量级，要么把 SGLang 对比拓扑改成同样 `GPU/Req=1` 的 serverful replica baseline，否则仅靠 dispatch 修复无法弥合 TP4 带来的 E2E 差距。

### 5. SGLang 原论文/官方文档校准后的对比设置

SGLang 原论文定位是高性能 runtime / programming system，核心优化包括 RadixAttention、structured decoding 和 runtime scheduling；论文实验目标是展示 SGLang runtime 在复杂 LM programs 上相对 Guidance、vLLM、LMQL 的吞吐/延迟优势，而不是 serverless LoRA lifecycle。官方文档同时明确支持：

- `TP`: tensor parallel，多 GPU 共同服务一个请求；
- `DP`: data parallel，在显存足够时更适合吞吐，并推荐用 router / data-parallel serving；
- `DP + TP`: 例如 `DP=2, TP=2` 共用 4 GPU。

因此，当前 `SGLang-TP4` 不是错误，但它代表的是“4 GPU model-parallel serverful upper bound”。如果把它作为唯一主对比，会把每请求 4 GPU 的 decode 加速和 FaaSLoRA 的 serverless lifecycle/LoRA preloading 贡献混在一起。

后续主实验应同时报告两条 SGLang 线：

- `SGLang-TP4`: serverful upper bound，说明极强 serverful runtime 在每请求 4 GPU 下的上限；
- `SGLang-DP4`: 4 个 TP1 replica，总 GPU 预算仍为 4，但每个请求只使用 1 GPU，和 FaaSLoRA 的 `GPU/Req=1 + scale-out` 拓扑可比。

当前已落地的实现策略：

- `run_sglang_fair_experiment.sh` 支持 `SGLANG_DATA_PARALLEL_REPLICAS=4`；
- DP 模式会启动 4 个 SGLang TP1 replica；
- `replay_openai_trace.py` 支持多 `base_url` round-robin；
- baseline summary 和 compare 表记录 `TP / GPUPerRequest / RuntimeGPUs / Topology`；
- 这样既不删除 `TP4`，也不人为改差 SGLang，而是补齐更贴近 FaaSLoRA 场景的 `GPU/Req=1` serverful baseline。
